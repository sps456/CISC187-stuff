Outside of the obvious problem of infinitely short paths caused by potential negative cycles, Dijkstra's path algorithm will fail with negative edge weights because it will treat a node as complete once it picks the smallest exit route from it. Take the example below:
<img width="378" height="406" alt="image" src="https://github.com/user-attachments/assets/215bd89f-56d7-43bf-9387-b646e7df49d4" />

The path A -> F -> E is the shortest path from A to E with a weight of 3, but Dijkestras algorithm will select the path A -> B -> C -> D -> E despite it having a weight of 4. This is because starting at node A, the algorithm will choose to travel to node B since it has a weight of 1 and the alternate path to node F has a weight of 9. Without the existance of negative edge weights, the route starting with weight 9 will never be faster than the route with a total weight of 4, so the algorithm will move onto node B and not return to node A. Even if the path gets shorter later because of negative edge weights, dijkestras algorithm will not traverse the "slow" path long enough to figure that out.
